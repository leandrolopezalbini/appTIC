🚨 Manejo de Errores ALTO IMPACTO

1. Try/Catch Vacíos o Insuficientes

🟡 Catch clauses con solo Logger.log()
Múltiples archivos

Ejemplos encontrados:

// ❌ 3_Utilidades.js - Linea 31 catch (e) { Logger.log("Error registrarHistorial: " + e); } // ❌ 3_Utilidades.js - Linea 53 catch (err) { Logger.log("Error al registrar acceso: " + err); } // ❌ 2_Consultas.js - Linea 36 catch (e) { Logger.log('Error Paradas Seguras: ' + e.message); } // ❌ 7_Triggers.js - Linea 43 catch (err) { return; } // ❌ 6_Notificaciones.gs - Linea 24 catch(e) { Logger.log("Error enviando mail: " + e); }
Problema: Los errores solo se registran en Logger (servidor), pero el FRONTEND no sabe qué pasó. Si una operación crítica falla, el usuario no recibe feedback.

Solución: Devuelve objetos de error estructurados y envía toasts al frontend:
try { // ... operación ... } catch (e) { Logger.log("Error registrarHistorial: " + e.message); // Devuelve estructura de error return { success: false, message: "Error al registrar: " + e.message, errorCode: "REG_HIST_ERROR" }; }
🟡 Errores no comunicados al Frontend
6_Notificaciones.js | notificarEvento()

Problema: Si MailApp.sendEmail() falla, solo se registra en Logger:

function notificarEvento(evento, detalles) { // ... lógica ... try { destinatarios.forEach(dest => { MailApp.sendEmail(dest, asunto, cuerpo); }); } catch(e) { Logger.log("Error enviando mail: " + e); // ❌ El usuario NO se entera } }
Estructura el error y déjalo disponible para que el frontend lo maneje:
function notificarEvento(evento, detalles) { const resultado = { success: true, notificaciones: 0, errores: [] }; try { destinatarios.forEach(dest => { try { MailApp.sendEmail(dest, asunto, cuerpo); resultado.notificaciones++; } catch(innerE) { resultado.errores.push({ correo: dest, error: innerE.message }); } }); if (resultado.errores.length > 0) { resultado.success = false; } } catch(e) { resultado.success = false; resultado.errores.push({ error: e.message }); } return resultado; }

3. Verificación Insuficiente de Datos Críticos

🟡 procesarSolicitudUnificada() no valida payloads
5_Operaciones.js | Linea 8

La función recibe data pero no verifica si contiene campos requeridos:

// ❌ No hay validación de data function procesarSolicitudUnificada(data) { try { // Si data.id, data.tipoAccion, etc. son undefined, pasa silenciosamente if (data.id && data.tipoAccion) {

function procesarSolicitudUnificada(data) {
  if (!data || typeof data !== 'object') {
    return { success: false, message: "Payload inválido" };
  }
  const requiredFields = ['id', 'tipoAccion', 'estado'];
  for (const field of requiredFields) {
    if (!data[field]) {
      return { success: false, message: `Campo requerido falta: ${field}` };
    }
  }
Mejores Prácticas para Errores:
  1. SIEMPRE devuelve objetos estructurados: { success: bool, message: string, data: *, errorCode: string }
  2. En Frontend: Intercepta respuestas fallidas y muestra M.toast()
  3. En Backend: Usa AppSheet/Google Apps Script's built-in error handling
  4. Loguea errors.log() además de Logger.log() para auditoría
  5. Nunca catches vacíos - SIEMPRE comunica algo al usuario

🟡 aplicarFormatoInsumos() - Múltiples setBackground() en bucle
5_Operaciones.js | Lineas 260-300
